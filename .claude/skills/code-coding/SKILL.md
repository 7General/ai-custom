# 编码阶段

## 阶段概述

编码阶段是将设计转化为可执行代码的关键阶段。通过规范化的编码流程和最佳实践，确保代码质量、可维护性和可测试性。本阶段采用测试驱动开发（TDD）和代码评审机制，从源头保证代码质量。

## 核心活动

### 1. 代码框架搭建

**目标**：搭建项目基础代码结构

**活动内容**：

- 项目目录结构创建
- 依赖管理配置和添加
- 基础类和接口定义
- 配置文件管理
- 日志框架集成

**输出产物**：

- 项目骨架代码
- 构建配置文件
- 依赖清单文件

### 2. 核心功能实现

**目标**：实现核心业务逻辑

**活动内容**：

- 业务逻辑编码
- 接口实现
- 数据模型实现
- 异常处理

**输出产物**：

- 功能源代码
- 配置文件
- 技术文档

### 3. 代码评审

**目标**：通过评审提升代码质量

**活动内容**：

- 代码规范检查
- 逻辑审查
- 安全审查
- 性能审查

**输出产物**：

- 评审记录
- 问题清单
- 改进建议

### 4. 设计一致性验证

**目标**：验证代码实现与设计文档的一致性

**活动内容**：

- 功能一致性检查
- 代码与设计对应检查
- 代码真实存在性检查
- 设计功能到代码行号映射
- 一致性报告生成

**输出产物**：

- 代码-设计一致性检查报告
- 功能映射清单
- 问题修复建议

## 编码流程

### 标准编码流程

```
需求理解
├─ 读取设计文档(design.md)
├─ 读取接口定义文档(api.md)
├─ 提取功能模块列表
├─ 提取接口定义列表
└─ 提取数据模型列表
↓
代码实现
├─ 接口实现
├─ 业务逻辑
└─ 错误处理
↓
代码评审
├─ 自我评审
└─ 同行评审
↓
设计一致性验证
├─ 功能一致性检查
├─ 代码与设计对应
├─ 代码真实存在性检查
└─ 生成映射报告
↓
质量门禁判断
    ├─ 各项指标达标？
    ├─ YES → 允许合并
    └─ NO  → 阻止合并
↓
设计文档生成（**必须输出产物**）
└─ 生成必要文档，docs/code/{feature}-code-review.md

```

## 使用示例

### 场景1：开始写代码

用户：用实现用户登录功能
AI：[触发z-code-skill]


🟢 第一步：实现最小代码

创建实现文件：auth.swift

要求：

1. 仅实现核心认证流程，暂不考虑异常处理和性能优化。
2. 优先保证功能可运行，避免过度设计。
3. 使用最少依赖完成实现，减少模块耦合。
4. 所有方法保留清晰的 TODO 标记，便于后续扩展。
5. 优先采用 Swift 原生能力，不引入第三方库。
6. 保持单一职责原则，一个文件只负责认证逻辑。
7. 为关键接口添加必要注释，说明输入、输出及用途。
8. 编写最小可验证示例，确保代码能够独立运行。


运行测试：通过 ✅（绿灯）


🔄 第二步：重构优化

优化点识别：

1. 降低模块耦合度，优化业务架构设计，提升代码可维护性。
2. 消除重复逻辑，抽取公共组件与工具类，提高开发效率。
3. 优化内存管理与页面性能，减少资源消耗和卡顿问题。
4. 提升代码可读性与健壮性，降低线上异常风险。
5. 推进组件化与配置化建设，提高业务开发复用率。

运行测试：通过 ✅

✅ 第三步：代码评审

自动评审结果：

✓ 代码规范：通过
✓ 类型提示：完整
✓ 文档字符串：完整
✓ 异常处理：合理

改进建议：

1. secret_key应从配置读取
2. 添加登录失败次数限制
3. 添加日志记录
