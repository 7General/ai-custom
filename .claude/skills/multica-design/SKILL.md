## 核心活动

### 1. 需求分析

**目标**：理解和澄清业务需求，定义验收条件

**活动内容**：
- 需求收集和整理
- 用户故事编写
- 验收条件定义（EARS格式）
- 需求优先级评估
- 技术可行性分析

**输出产物**：
- 需求规格文档
- 用户故事列表
- 验收条件清单

### 2. 架构设计

**目标**：设计系统整体架构和技术选型

**活动内容**：
- 系统架构设计
- 技术栈选择
- 模块划分
- 依赖关系定义
- 架构决策记录（ADR）

**输出产物**：
- 架构设计文档
- 架构图
- 技术选型说明

### 3. 接口实现

**目标**：定义接口

**活动内容**：
- API接口定义
- 请求响应格式定义
- 错误码设计
- 接口文档编写

**输出产物**：
- API接口文档
- 接口契约文件

### 4. 数据模型设计

**目标**：设计实体`Entity`的字段

**活动内容**：
- 创建实体对象`Entity`

**输出产物**：
- 数据模型文档



## 强制产物规则

**每次设计完成后，必须生成以下文件（缺一不可）：**

| 产物 | 路径 | 说明 |
|------|------|------|
| 技术设计文档 | `docs/design/{feature}-design.md` | 需求、架构、数据模型 |
| API 接口文档 | `docs/api/{feature}-api.md` | 接口路径、参数、响应、错误码 |

> ⚠️ 即使接口简单，也必须生成 `docs/api/` 文件。不允许只生成设计文档而跳过接口文档。

### API 接口文档必须包含

- 接口列表（方法 + 路径 + 说明）
- 每个接口的：请求参数表、请求示例 JSON、响应示例 JSON（成功 + 失败）
- 错误码表及客户端处理策略
- 本地存储 Key 对照表（如有）

## 设计流程

### 标准设计流程

```
需求输入
    ↓
需求分析
    ├─ 需求澄清
    ├─ 用户故事编写
    └─ 验收条件定义
    ↓
架构设计
    ├─ 系统架构设计
    ├─ 模块划分
    └─ 技术选型
    ↓
详细设计
    ├─ 接口实现
    ├─ 数据模型设计
    └─ 非功能设计
    ↓
设计评审
    ├─ 技术评审
    ├─ 安全评审
    └─ 性能评审
    ↓
设计文档生成（必须同时生成两个文件）
    ├─ docs/design/{feature}-design.md
    └─ docs/design/{feature}-api.md
```

## 设计评审清单

### 接口评审

- [ ] 错误处理是否区分网络错误 / 业务错误码，并有对应 UI 提示
- [ ] 接口是否通过 OPBNetworkHelper（或项目统一网络层）发起，禁止直接使用 URLSession 
- [ ] 接口命名是否清晰
- [ ] 错误处理是否完整
- [ ] 是否有版本控制
- [ ] 是否考虑向后兼容

### 数据模型评审

- [ ] Entity / Model 字段是否完整，类型是否合理（避免用 Any / AnyObject）
- [ ] 是否明确区分持久化存储（OPBStorageHelper）与临时缓存（内存）
- [ ] 新增 UserDefaults Key 是否评估 logout 时是否需要清除，并更新白名单

### 安全评审

- [ ] 网络请求是否走 HTTPS，是否禁止 HTTP 明文（ATS配置）
- [ ] 用户登出时是否清除缓存（UserDefaults白名单 + OPBStorageHelper）
