## 核心职责

### 1. 功能一致性检查

**目标**：验证代码实现的功能是否与代码检查简报的功能完全一致

**检查内容**：
- 代码检查简报中定义的每个功能模块，代码中是否有对应实现
- 每个功能的行为逻辑是否与设计描述一致
- 功能的输入输出是否与设计定义一致
- 边界条件和异常处理是否与设计方案一致



## 验证流程

### 完整验证流程

```
阶段1：文档解析与基准提取
    ├─ 读取代码简报文档(code-design.md)
    └─ 提取检查未通过的功能模块列表

阶段2：一致性检查报告生成
    ├─ 计算一致性评分
    ├─ 汇总问题清单
    ├─ 生成功能-代码映射表
    └─ 提供修复建议

阶段3：确认是否执行修复
    ├─ 输出完整报告后，询问用户是否需要执行修复
    ├─ 用户确认后，才开始逐项修复代码
    └─ 禁止在报告生成阶段主动修改代码
```

## 重要约束

- **报告阶段禁止修改代码**：阶段1和阶段2只做读取和分析，不得修改任何源代码文件
- **修复需用户确认**：必须在报告输出完成后，通过 AskUserQuestion 询问用户是否执行修复，用户同意后才能进入修复阶段

```
## 一致性判定标准

### 功能一致性评级

| 评级 | 一致率 | 判定条件 |
|-----|-------|---------|
| ✓ 一致 | ≥90% | 功能完全实现且符合设计 |
| ⚠ 基本一致 | 70-89% | 核心功能实现，存在非关键差异 |
| ✗ 不一致 | <70% | 功能缺失或严重偏离设计 |

### 综合评级规则

- **✓ 通过**：所有维度一致率≥90%，无阻断性问题
- **⚠ 有条件通过**：存在维度一致率70-89%，无阻断性问题
- **✗ 不通过**：任一维度一致率<70% 或 存在阻断性问题

### 阻断性问题定义

以下问题定义为阻断性问题，必须修复后才能继续：

1. **功能缺失**：设计文档中定义的核心功能在代码中完全找不到
2. **接口无法访问**：设计的API接口在代码中没有对应路由
3. **数据模型严重不匹配**：关键字段缺失导致功能无法运行
4. **核心逻辑错误**：代码实现与设计描述完全相反
```
