# 代码审查

## 规范来源

| 规范 | 参考文件 |
|------|---------|
| 编码通用规范（必须） | `rules/编码规范.md` |
| 组件目录结构 | `rules/组件结构.md` |
| SnapKit 约束（含UI布局时） | `snapkit-skill/SKILL.md` |
| 网络请求（含请求代码时） | `request-send-skill/SKILL.md` |
| 主题颜色表 | `code-skill_new/reference/theme-color-map.md` |

> 审查前**必须读取** `rules/编码规范.md` 和被审查的代码文件。

---

## 审查流程

```
1. 读取规范文件 + 代码文件
2. 编码规范检查
3. 静态分析（复杂度、重复、潜在Bug）
4. 设计一致性检查（如有设计文档）
5. 输出审查结果
```

---

## 检查项

### 1. 编码规范检查

逐项对照 `rules/编码规范.md`，输出结果表：

| 编号 | 规范要求 | 代码现状 | 位置 | 状态 |
|------|---------|---------|------|------|
| R-001 | 懒加载闭包内用 `it` | ... | :行号 | ✓/✗ |
| R-002 | 禁止 `_` 开头变量/方法 | ... | :行号 | ✓/✗ |
| R-003 | 普通变量在 viewDidLoad() 前 | ... | :行号 | ✓/✗ |
| R-004 | 继承正确基类 | ... | :行号 | ✓/✗ |
| R-005 | 有 deinit 方法 | ... | :行号 | ✓/✗ |
| R-006 | 颜色使用 theme_ 前缀 | ... | :行号 | ✓/✗ |

### 2. Swift 静态分析

- **强制解包**：检查 `!` 使用
- **循环引用**：检查闭包中 `[weak self]`
- **guard 使用**：是否合理使用 guard 提前退出
- **主线程阻塞**：耗时操作是否在主线程
- **重复代码**：相似逻辑是否可复用
- **函数长度**：单个函数是否过长
- **单一职责**：类/函数职责是否清晰

### 3. 设计一致性检查（有设计文档时）

如果存在 `docs/design/` 下的设计文档，对照检查：

| 检查维度 | 说明 |
|---------|------|
| 功能一致性 | 设计文档中的功能是否都有代码实现 |
| 接口对应性 | 设计的API是否与代码路由一致 |
| 数据模型对应性 | Entity字段是否与设计一致 |
| 代码真实存在性 | 设计中提到的文件/函数是否真实存在 |

### 4. 未找到的主题颜色

列出代码中使用了但在 `reference/theme-color-map.md` 中找不到的颜色值。

## 输出格式

直接在终端输出审查结果，格式：

```
📋 代码审查结果

✅ 通过项：X 项
⚠️ 警告项：X 项
❌ 问题项：X 项

--- 问题详情 ---

1. [❌/⚠️] 问题描述
   位置: 文件:行号
   建议: 修复方式

--- 编码规范一致性 ---

(规范检查表)

一致率：X/Y (Z%)

--- 未找到的主题颜色 ---

(颜色列表，如无则跳过)
```

> 无需强制生成文档文件。用户需要时再生成。
